# S1: 주말아이 1차 실행 후보 3개 선정 및 7일 실험 설계

작성 시각: 2026-05-29 19:51 KST

## 결론 먼저

1차 실행은 **1순위 후보만 바로 구현**하는 편이 안전합니다.

- 1순위: 핵심 롱테일 행사 허브 신뢰도·색인 실험
  - 바로 실행합니다.
  - 기존 행사 데이터, 가이드, sitemap, Search Console 흐름을 살립니다.
  - AdSense 신청 전 필요한 404·thin page·행사 품질 문제를 먼저 줄입니다.
- 2순위: 날씨·미세먼지 기반 실내 대안 페이지
  - 1순위가 통과한 뒤 붙입니다.
  - 검색 의도는 좋지만, 날씨·대기질 데이터가 빠르게 바뀌므로 운영 자동화가 필요합니다.
- 3순위: 계절 공공시설 허브
  - 여름 물놀이장처럼 한 주제만 작게 검증합니다.
  - 시즌 검색 수요는 있지만 공식 정보 변경 리스크가 있어 1순위보다 뒤입니다.

7일 안에 매출을 목표로 잡지 않습니다. 7일 목표는 색인·품질·운영 가능성을 확인하는 것입니다. 수익 추정은 R1의 보수적 Page RPM 계산식만 참고하고, 보장 수익처럼 말하지 않습니다.

## 선정 기준

- 태훈님이 직접 영업하거나 계약하지 않아도 되는가.
- 공공데이터·검색 유입·자동 생성·정적 배포에 잘 맞는가.
- 7일 안에 측정 가능한 지표가 나오는가.
- 개인정보, 저작권, 공공데이터 출처, 안전·무료 오안내 리스크가 낮은가.
- 현재 주말아이 자산을 해치지 않고 오히려 AdSense·색인 준비도를 높이는가.

## 1순위: 핵심 롱테일 행사 허브 신뢰도·색인 실험

### 한 줄 정의

`서울 아이랑 무료 행사`, `이번 주말 아이랑`, `구별 어린이 행사` 검색 유입을 받을 기본 허브를 신뢰 가능한 페이지 묶음으로 고도화합니다.

### 포함 범위

- 존재하지 않는 URL이 홈페이지 200으로 응답하는 soft 404 문제 제거
- 홈 첫 화면의 어린이·가족 relevance score 개선
- 행사 상세 페이지의 예약·요금·연령 확인 문구 보강
- `/about/`, `/contact/`, `/sources/`, `/privacy/` 신뢰 페이지 보강
- 기존 가이드 5개와 핵심 랜딩의 내부링크 정리
- sitemap, canonical, robots, lastmod 재검증
- 운영 헬스체크와 주간 리포트 자동화 연결

### 왜 1순위인가

- R1의 1번과 6번을 합친 형태라 기존 데이터 파이프라인과 가장 잘 맞습니다.
- R2가 지적한 soft 404와 thin page 문제를 해결하지 않으면 AdSense·색인 실험이 흔들립니다.
- A1의 운영 헬스체크와 바로 연결되어 조용한 실패를 줄일 수 있습니다.
- 새 외부 계정이나 제휴 승인을 기다리지 않아도 시작할 수 있습니다.
- 사용자에게 보이는 수익 문구를 넣지 않아도 사이트 품질 지표를 만들 수 있습니다.

### 7일 실험 계획

- 1일차: 범위 고정
  - B1은 404, relevance score, 행사 상세 보강, 신뢰 페이지 보강 테스트를 먼저 작성합니다.
  - A2는 라이브 헬스체크 항목과 상태 파일 스키마를 확정합니다.
  - 기준값을 기록합니다: 로컬 행사 수, sitemap URL 수, `/seoul/free/`, `/seoul/this-weekend/`, `/seoul/indoor/` 행사 수.

- 2일차: soft 404와 품질 정렬 구현
  - `404.html` 생성과 Cloudflare Pages missing route 처리를 반영합니다.
  - `청소년 이상 성인`, `성인 동반에 한해 가능` 같은 약한 가족 적합도 문구를 하향합니다.
  - 홈 첫 노출은 strong-match 중심으로 정렬합니다.

- 3일차: 행사 상세 페이지 보강
  - 예약·요금·연령·장소 확인 포인트를 행사 필드 기반으로 다르게 표시합니다.
  - 애매한 무료·연령 정보는 `확인 필요`로 낮춥니다.
  - 공식 링크와 방문 전 확인 안내를 유지합니다.

- 4일차: 가이드·신뢰 페이지 보강
  - 기존 가이드 5개를 검색 의도별 실용 안내로 보강합니다.
  - `/sources/`에는 데이터 출처, 갱신 방식, 오분류 정정 원칙을 명확히 적습니다.
  - `/privacy/`에는 향후 AdSense·Analytics 도입 시 쿠키·광고 식별자 안내가 필요하다는 구조를 반영합니다.
  - `/contact/`는 공개 가능한 연락 채널이 없으면 `태훈님 확인 필요` 항목으로 분리합니다.

- 5일차: 빌드·테스트·샘플 감사
  - `python -m pytest tests -q`를 통과시킵니다.
  - `bash scripts/build.sh`를 통과시킵니다.
  - 행사 50건 표본에서 무료·연령·예약 오판 후보를 기록합니다.
  - sitemap XML 파싱과 percent-encoding을 확인합니다.

- 6일차: 배포 후 라이브 검증
  - 없는 URL이 404로 응답하는지 확인합니다.
  - `/`, `/robots.txt`, `/gsc-sitemap.xml`, `/sitemap-index.xml`, `/sitemap.xml`이 정상인지 확인합니다.
  - 대표 상세 페이지와 핵심 랜딩의 canonical이 `https://jumalikids.com/` 계열인지 확인합니다.

- 7일차: Search Console 사용자 작업과 운영 리포트 확인
  - 태훈님이 Search Console URL-prefix 속성 `https://jumalikids.com/`에서 `sitemap-index.xml` 또는 `gsc-sitemap.xml`을 확인합니다.
  - 홈, `/seoul/free/`, `/seoul/this-weekend/`, 대표 행사 상세 1~2개 URL 검사를 기록합니다.
  - A2 자동 리포트가 상태, 카운트, 주의 항목, 태훈님 직접 작업을 짧게 요약하는지 확인합니다.

### 성공 기준

- 없는 URL 3종 이상이 라이브에서 404로 응답합니다.
- 테스트와 빌드가 통과합니다.
- sitemap XML이 Googlebot UA 기준 파싱됩니다.
- 홈 첫 화면에 성인·청소년 중심 행사가 우선 노출되지 않습니다.
- 대표 행사 상세 페이지가 필드별로 차별화된 안내를 가집니다.
- `/about/`, `/sources/`, `/privacy/`, `/contact/`가 단순 placeholder처럼 보이지 않습니다.
- 50건 표본 점검에서 `무료`, `어린이 적합`, `예약 필요`를 확정적으로 잘못 말하는 사례가 발견되면 즉시 수정하거나 `확인 필요`로 낮춥니다.
- Search Console에서 최소 3개 핵심 URL이 크롤링 가능 상태로 확인되거나, 사용자가 확인해야 할 막힌 지점이 명확히 기록됩니다.

### 중단 또는 보류 기준

- Cloudflare Pages에서 missing route가 계속 홈페이지 200으로 남습니다.
- strong-match 행사가 너무 적어 핵심 랜딩이 빈 페이지처럼 보입니다.
- 콘텐츠 보강이 실제 정보 없이 문장만 늘리는 방식이 됩니다.
- 공개 연락 채널이 없어 `/contact/` 보강이 막히고 AdSense 신청 판단이 불가능합니다.
- 자동 리포트가 실패를 조용히 삼키거나, 정상/주의/실패 기준을 설명하지 못합니다.

## 2순위: 날씨·미세먼지 기반 실내 대안 페이지

### 한 줄 정의

`비오는날 서울 아이랑 실내`, `미세먼지 나쁨 아이랑 갈만한 곳`처럼 상황형 검색 의도를 받는 랜딩을 만듭니다.

### 왜 2순위인가

- 검색 의도가 구체적이고 부모 입장에서 바로 쓸 수 있습니다.
- 기존 `/seoul/indoor/`와 행사 태그를 재사용할 수 있습니다.
- 기상청·에어코리아 데이터를 붙이면 자동화 가치가 큽니다.
- 다만 날씨·대기질은 빠르게 변하므로, 1순위의 사이트 신뢰도와 운영 리포트가 먼저 있어야 합니다.

### 7일 실험 계획

- 1일차: API 키 필요 여부와 호출 제한을 확인합니다.
- 2일차: 외부 API 없이도 동작하는 `/guides/rainy-day-indoor-kids-seoul/`와 `/seoul/indoor/` 개선안을 확정합니다.
- 3일차: 비·미세먼지·실내 키워드별 랜딩 3~6개만 만듭니다.
- 4일차: 날씨·대기질 문구는 안전 보장처럼 쓰지 않고 `방문 전 공식 안내 확인` 문구를 넣습니다.
- 5일차: 실제 날씨 API가 붙으면 캐시와 실패 fallback을 둡니다.
- 6일차: sitemap, canonical, 내부링크를 확인합니다.
- 7일차: Search Console URL 검사와 클릭 가능한 내부 링크 흐름을 기록합니다.

### 성공 기준

- 상황형 랜딩 3개 이상이 실제 행사 데이터와 연결됩니다.
- 외부 API 실패 시에도 페이지가 깨지지 않습니다.
- 건강·안전 보장처럼 보이는 표현이 없습니다.
- 실내 행사 수, 업데이트 시각, 공식 링크가 표시됩니다.

### 중단 기준

- API 키나 약관 문제로 자동 갱신이 불안정합니다.
- 실내 행사 데이터가 너무 적어 페이지가 얇아집니다.
- 미세먼지·날씨 상태를 정확히 설명하지 못해 사용자 오해가 생길 수 있습니다.

### 필요한 후속 카드

## 3순위: 계절 공공시설 허브

### 한 줄 정의

여름 물놀이장, 어린이날, 겨울 눈썰매장처럼 시즌 검색어가 강한 주제를 하나만 골라 공식 링크 기반 페이지 묶음으로 검증합니다.

### 왜 3순위인가

- 시즌 검색 수요가 뚜렷합니다.
- 10개 내외 페이지로 작게 시작할 수 있습니다.
- 공공데이터·지자체 공식 링크와 잘 맞습니다.
- 다만 운영기간, 요금, 휴장일이 자주 바뀌므로 마지막 확인일과 만료 처리가 꼭 필요합니다.

### 7일 실험 계획

- 1일차: 한 주제만 정합니다. 현재 시점에서는 여름 물놀이장 또는 여름 무료 체험이 후보입니다.
- 2일차: 공식 링크 10개 이하를 seed 파일로 정리합니다.
- 3일차: 운영기간, 요금, 대상, 위치, 공식 링크, 마지막 확인일 필드를 정합니다.
- 4일차: 시즌 허브 1개와 상세 5~10개를 생성합니다.
- 5일차: 지난 시즌 정보가 노출되지 않도록 만료 기준을 테스트합니다.
- 6일차: sitemap 포함 기준과 `lastmod`를 확인합니다.
- 7일차: Search Console URL 검사와 사용자 오해 가능성을 리뷰합니다.

### 성공 기준

- 공식 링크가 있는 시즌 항목 5개 이상을 만들 수 있습니다.
- 각 페이지에 마지막 확인일과 공식 확인 안내가 있습니다.
- 기간이 지난 항목은 자동으로 숨기거나 낮춥니다.
- 운영기간·요금·대상 정보를 확정적으로 과장하지 않습니다.

### 중단 기준

- 공식 정보가 흩어져 있어 수동 확인 비중이 너무 큽니다.
- 운영기간·요금 변경 가능성이 높아 자동화 없이 유지하기 어렵습니다.
- 시즌이 지나 검색 수요가 약해졌습니다.

### 필요한 후속 카드

지금은 만들지 않습니다. 1순위 검수 후 실행할 때 생성합니다.

- B3 후보 — `jumalibuilder`: 시즌 seed 파일, 허브/상세 페이지, 만료 테스트 구현
- A4 후보 — `jumaliauto`: 공식 링크 상태, 만료일, 시즌 항목 수 급감 알림 추가
- Q3 후보 — `jumalireview`: 공식 출처·요금·운영기간 정확성 검수

  - 프로그램 데이터가 지역별 홈페이지에 흩어져 있습니다.
  - 공식 API/RSS가 없는 곳은 무단 크롤링 리스크가 있어 이번 조건과 맞지 않습니다.

- 전국 어린이 놀이터 안전·편의 지도
  - 위치 데이터는 가능하지만 실제 안전·그늘·화장실 같은 현장 정보는 자동화가 어렵습니다.
  - `안전` 표현은 보증처럼 보일 수 있어 리스크가 큽니다.

- 방학 무료 체험·캠프 시즌 허브
  - 시장성은 있지만 소스가 다양하고 예약 마감이 빨라 운영 난이도가 높습니다.
  - 3순위 계절 허브가 통과하면 그다음 주제로 확장합니다.

- 주간 자동 리포트·뉴스레터
  - 재방문에는 좋지만 이메일/텔레그램 구독 수집, 동의, 수신거부 처리가 필요합니다.
  - 초기 트래픽이 낮으면 7일 지표가 거의 나오지 않을 수 있습니다.

- 공공데이터 미니사이트 스타터킷
  - 비대면 디지털 상품이지만 고객이 부모가 아니라 제작자입니다.
  - 설치 지원 요구가 생기면 자동화 조건과 충돌할 수 있습니다.

- 아이랑 당일치기 루트·소요시간 플래너
  - 이동시간, 지도 API 약관, 휴무일 정확도 리스크가 큽니다.
  - 장소형 페이지로 축소 검증하기 전에는 보류합니다.

## A2에 넘길 구체 작업 정의

A2는 1순위 후보 운영 자동화만 만듭니다.

작업 목표:

- 사이트가 조용히 망가지지 않게 합니다.
- 1순위 실험의 핵심 지표를 매일/매주 확인합니다.
- 태훈님이 직접 해야 하는 Search Console·AdSense 작업을 자동화 작업과 분리합니다.

필수 점검 항목:

- GitHub Actions 최근 scheduled run 상태와 시각
- `/`, `/robots.txt`, `/gsc-sitemap.xml`, `/sitemap-index.xml`, `/sitemap.xml` HTTP 상태
- sitemap XML 파싱 성공 여부
- 없는 URL이 404로 응답하는지
- live free/this-weekend/indoor 행사 수
- sitemap URL 수 급감 여부
- 홈페이지 또는 핵심 랜딩의 canonical이 `https://jumalikids.com/` 계열인지
- strong-match 또는 adult/broad-match 의심 항목 샘플
- 무료/요금 충돌 의심 항목 샘플
- `최종 업데이트`가 오래됐는지

구현 후보:

- Hermes cron 기반 daily silent check
  - 실패 또는 주의 조건이 있을 때만 알립니다.
- Hermes cron 기반 weekly report
  - 정상이어도 주 1회만 요약합니다.
- 상태 파일
  - 이전 정상 카운트, 마지막 성공 시각, sitemap URL 수를 저장합니다.
  - 프로필별 안전한 state 경로를 사용합니다.
- 문서
  - repo 안에는 자동화 동작과 태훈님 직접 작업을 설명하는 짧은 운영 문서를 남깁니다.

알림 기준:

- 실패
  - 최근 Actions run이 success가 아님
  - 최근 scheduled run이 36시간 이상 없음
  - 핵심 URL이나 sitemap이 200이 아님
  - sitemap XML 파싱 실패
  - 없는 URL이 200 홈페이지로 응답
  - `/seoul/free/` 행사 수가 0
  - sitemap URL 수나 MVP 행사 수가 이전 정상 대비 50% 이상 감소
- 주의
  - 이번 주말 행사 수가 0이지만 전체 무료 행사는 남아 있음
  - 공식 URL 누락 비율이 높음
  - adult/broad-match 의심 항목이 홈 우선순위에 섞임
  - 최종 업데이트가 3일 이상 오래됨

태훈님 직접 작업으로 분리할 것:

- Search Console 소유권 확인
- Search Console URL 검사와 색인 생성 요청 클릭
- sitemap 제출 상태 확인
- AdSense 신청, 사이트 연결, 정책 동의
- 공개 연락 이메일 또는 문의 채널 결정

검증 방식:

- 스크립트 dry-run 결과를 남깁니다.
- cron 생성 시 job id, schedule, prompt 또는 script 경로를 남깁니다.
- 실패 조건을 강제로 하나 이상 시뮬레이션하거나, 최소한 없는 URL 404 점검 결과를 포함합니다.

완료 방식:

- 자동화가 실제로 생성됐으면 검증 결과를 함께 남깁니다.
